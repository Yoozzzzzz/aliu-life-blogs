## 0. 目标与边界

**目标**：在 **BFF（`server/`）** 侧提供与示例软件「多策略按优先级尝试」同构的能力，使 VIP/灰版权益在官方链失败时，仍可按配置依次尝试 **UNM 以外/以内** 的可插拔策略，并具备 **缓存、失败短窗跳过、统一响应格式**，供 `app` 的 `fetchPlayableUrl` 一条 HTTP 拉齐或分步拉齐。

**架构约束**（静渊与 Electron 版差异）：

- 无 `window.api.unblockMusic`；**策略须在 Node BFF 内实现**，禁止把第三方解析密钥或可滥用端点明文下发给浏览器。
- 现有 Alger 仍由 BFF 内联启动；策略可 **调用本机 Alger**、**出站 HTTP**（自定义 API、GD、B 站搜索等），需统一 **超时、日志脱敏、错误码**。

## 1. IN（应交付）

### 1.1 服务端：策略框架

| 编号 | 事项 | 交付要点 |
|------|------|----------|
| S1 | 策略接口 | 定义统一签名，例如：`{ name, priority, canHandle(ctx), resolve(ctx) -> { url, meta } \| null }`；`ctx` 至少含 `songId`、**song/detail 元数据**（与现 UNM 元数据同源）、可选 `cookie`、请求级 `traceId`。 |
| S2 | 策略注册表 | 配置驱动：**启用列表 + 顺序**与环境变量默认值；可按环境关闭整类策略（与 `NCM_UNBLOCK_ENABLE` 协调，避免语义重复冲突）。 |
| S3 | 编排器 | 按优先级 **串行** 尝试（与示例一致）；单策略 **超时**（可配置，建议默认 ≤15s）；任一成功即返回；全失败返回结构化错误（见 AC）。 |
| S4 | UNM 策略 | 将当前 `/jy/unblock/url` 内逻辑 **收编为策略之一**（如 `unblock`），与其它策略共用元数据与缓存键规则。 |

### 1.2 服务端：策略实现（与示例对齐的「做满」范围）

> 下列每项可作为子任务分支；**首发里程碑**可只含 S4 + 下表中加粗项，其余在确认需求与合规后迭代。

| 编号 | 策略 | 交付要点 |
|------|------|----------|
| P1 | **UNM（已有，重构接入）** | `match` + `NCM_UNBLOCK_SOURCES`；song/detail 已有时保留。 |
| P2 | **自定义 API（Custom）** | 与 AlgerMusicPlayer 类似：可配置 **基础 URL、鉴权头（仅服务端环境变量）**、请求模板；**禁止**把密钥写进前端仓库。 |
| P3 | B 站音频（Bilibili） | 出站搜索 + 取音；注意 robots/频率；**默认关闭或限流**，由 env 控制。 |
| P4 | GD 音乐台 / 在线解析 | 若对接第三方 HTTP API：超时、失败缓存、用户 agent（文档化）。 |
| P5 | 落雪脚本（Lx） | **可选**：Node 侧执行脚本或 HTTP 桥接；若成本过高，任务书 **第二期** 再开，本任务 IN 可为「预留接口 + OUT 实现」。 |

### 1.3 缓存与失败退避

| 编号 | 事项 | 交付要点 |
|------|------|----------|
| C1 | 成功缓存 | 内存或 **可选** 文件/redis（首版内存即可）：键含 `songId` + **策略集合哈希** + 会话桶（若与 Cookie 档位相关）；TTL 可配置（建议成功 15–30min，与现前端 5min 协调：或由 BFF 统一缓存、前端缩短重复请求）。 |
| C2 | 失败短窗 | 单策略、单曲级 **短 TTL 内跳过**（示例：1min），避免播放失败时狂刷。 |
| C3 | 显式失效 | 与现前端 `invalidatePlayUrlCache` 对齐：提供 **BFF 侧** 清缓存约定（可选路径或文档说明「仅前端 TTL/换 id」）。 |

### 1.4 HTTP 契约与配置

| 编号 | 事项 | 交付要点 |
|------|------|----------|
| A1 | 统一入口 | **推荐**：`GET|POST /jy/resolve/play-url`（或保留 `GET /jy/unblock/url` 为兼容层转发新编排器）；响应 JSON 统一：`code`、`url`、`source`（策略名）、`qualityHint`（可选）、`message`。 |
| A2 | 环境与文档 | `server/README.MD` 增加：**策略开关、顺序、超时、自定义 API 变量**；更新 `docs/后端接口文档.MD` 中对应章节。 |
| A3 | `npm run dev:api` | 根脚本或 `server/package.json` 中示例 env 与生产说明分离（开发默认值不提交密钥）。 |

### 1.5 前端（最小改动）

| 编号 | 事项 | 交付要点 |
|------|------|----------|
| F1 | 解析入口 | `fetchPlayableUrl` 在官方链失败后 **调用新 BFF 编排接口**（可一层请求完成多策略，避免前端轮询多个端点）。 |
| F2 | 无感 | 不改变现有 UI 原则；失败文案仍走 `playBlockHint`；替链音质展示可与现 `unblock:` 标签一致或扩展为策略名。 |

## 3. 验收标准（AC）

1. **编排**：在 `NCM_*` 合理配置下，关闭官方 URL 的模拟场景（或变灰单曲）仍能 **经 BFF 自动多策略** 得到可播 `url`，且顺序与配置一致（日志可查策略名，生产可脱敏）。
2. **兼容**：未开启任何扩展策略时，行为与当前「仅 UNM」等价；旧客户端若仍调 `/jy/unblock/url`，**不破**（返回格式不变或 301 文档说明）。
3. **安全**：自定义 API 等密钥 **仅进程环境变量**；BFF 日志不打印完整第三方 token。
4. **稳定**：单策略超时不会拖死整请求；全失败返回明确 `code/message`，前端不误判为网络 502 吞掉。
5. **文档与联调**：`server/README.MD` + `docs/后端接口文档.MD` 与实现一致；`npm run dev` 路径可验至少一条「官方失败 → BFF 成功」用例。

---

## 5. 风险与合规

- 多策略出站增加 **版权与平台 ToS** 风险；部署与是否开启由维护者自担，与 `server/README.MD` 声明一致。
- B 站 / 第三方 API **频控与封禁**；须限流与默认关闭策略。
- 缓存 **错误 URL** 会导致短时难恢复；失败缓存 TTL 须短，成功缓存须可在换源配置时 **键版本化**。
